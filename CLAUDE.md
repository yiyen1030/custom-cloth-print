# CLAUDE.md

本文件提供給 Claude Code 在此專案中工作時的背景說明。

## 專案概述

客製化商品網站的「印圖編輯器」— 讓客戶上傳圖案，在商品上拖拉/縮放/旋轉，指定要印刷的位置，
系統換算出精確座標與尺寸，供後端合成送印檔或交給印刷廠。

**目前狀態：** 已有一個可運作的前端原型（單一 HTML 檔），尚未串接任何後端、儲存或真實商品圖。
這是「概念驗證」階段，接下來要把它整合進實際網站並補上後端。

## 現有檔案

- `tshirt-customizer.html` — 前端原型，單檔 HTML + Fabric.js（CDN 引入，未打包），可直接用瀏覽器開啟測試。

## 技術選型與原因

- **Fabric.js 5.3.0**：用來做「圖片上傳後可拖拉/縮放/旋轉，並限制在指定範圍內」的 canvas 互動。
  選它是因為社群成熟、API 直接、能匯出物件座標，不需要自己刻手勢與變形邏輯。
- 目前用 CDN `<script>` 引入，尚未走 npm/打包流程 — 若專案要整合進正式前端（React/Vue 等），
  建議改用 `npm install fabric` 並依框架封裝成元件。

## 核心邏輯（重要，之後改動要維持這個原則）

1. **比例尺（SCALE）**：畫面像素與實際公分的換算比例（目前是 `5px = 1cm`），寫死在程式碼常數裡。
   這個值必須和「後端合成/印刷檔產生」用的比例尺完全一致，否則座標會對不上。
2. **可印刷範圍（AREA）**：一個矩形區域（目前是示範用的座標），代表商品上實際能印刷的範圍。
   - 圖案物件的移動/縮放都會被 `constrainToArea()` 限制在這個範圍內，不會超出邊界。
   - 不同商品、不同角度（正面/背面/袖子）都需要各自一組 `AREA` 座標，目前程式只寫了「T 恤正面」一組。
3. **輸出資料格式**：按下「產生列印資料」會輸出 JSON，欄位為：
   ```json
   {
     "order_ref": "...",
     "garment": "tshirt",
     "view": "front",
     "print_area_cm": { "width": 30, "height": 40 },
     "design": {
       "filename": "...",
       "position_cm": { "x": 0, "y": 0 },
       "size_cm": { "width": 0, "height": 0 },
       "rotation_deg": 0
     }
   }
   ```
   這份 JSON 是前後端的合約，後端要用同一份定義去解析。

## 待辦事項（依優先順序）

1. **換成真實商品圖**：目前商品輪廓是用 `fabric.Path` 手畫的簡化 SVG，需要換成實際商品的正式底圖
   （建議用透明背景 PNG 或 SVG，並重新量測、填入正確的 `AREA` 座標）。
2. **多商品 / 多角度支援**：把 `AREA`、商品底圖、比例尺抽成設定檔（例如 `products.json`），
   讓每個商品（T恤、帽子、馬克杯…）× 每個角度（正面、背面…）各自對應一組設定，而不是寫死在程式碼中。
3. **圖片上傳串接儲存服務**：目前圖片是用 `FileReader` 讀成 base64 存在瀏覽器記憶體，
   還沒上傳到伺服器。需要接上傳 API（例如 S3 / GCS presigned URL），JSON 輸出的 `filename`
   要換成實際的檔案 URL。
4. **後端合成邏輯**：需要一支服務（建議 Python + Pillow，或 Node.js + sharp）依照前端傳來的 JSON
   座標，把原始高解析度圖案疊加到商品的送印檔模板上，輸出可交印刷廠的檔案。
   **關鍵要求**：這支服務用的比例尺、原點（左上角）定義，必須與前端完全一致。
5. **表單送出流程**：目前「產生列印資料」只是把 JSON 印在畫面上，還沒有實際送出到後端 API
   （送出前要接使用者登入狀態、購物車、訂單編號等）。
6. **響應式與行動裝置**：目前主要以桌面尺寸設計，觸控裝置上的縮放/旋轉手把可能需要調整
   （Fabric.js 有內建行動支援，但角落控制點大小、觸控熱區建議另外測試調整）。
7. **驗證與錯誤處理**：目前沒有檔案格式/大小限制、上傳失敗處理、圖案解析度過低警告等，
   實際上線前需要補上。

## 已知限制（暫不處理，但要讓下一個人知道）

- 目前的「合成預覽」只是平面疊圖，沒有模擬衣服皺褶/光影，如果之後要做擬真預覽，
  是另外一條技術路線（例如額外接 AI 圖像生成 API 做視覺效果，跟這裡的精確定位邏輯是分開的兩件事，
  AI 生成圖不能拿來當送印檔，只能當作展示用預覽圖）。
- 沒有多語系、沒有無障礙（a11y）測試。
- 沒有單元測試。

## 開發慣例

- 目前所有邏輯都在單一 HTML 檔的 `<script>` 內，尚未模組化。若要擴充功能，建議先決定
  前端框架（是否導入 React/Vue）與建置工具（Vite 等），再把邏輯拆成元件，而不是繼續在單檔裡疊加。
- 顏色、字體等視覺樣式目前直接寫在 `<style>` 的 CSS 變數裡（`--paper`、`--ink`、`--thread` 等），
  之後如果有品牌設計系統（design tokens），這裡應該替換成對應的變數來源。
