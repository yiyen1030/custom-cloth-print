# CLAUDE.md

本文件提供給 Claude Code 在此專案中工作時的背景說明。

## 專案概述

客製化商品網站的「印圖編輯器」— 讓客戶上傳圖案，在商品上拖拉/縮放/旋轉，指定要印刷的位置，
系統換算出精確座標與尺寸，供後端合成送印檔或交給印刷廠。

**目前狀態：** 已有一個可運作的前端原型（HTML + CSS + JS 三檔 + 外部商品圖），尚未串接任何後端或
儲存服務。這是「概念驗證」階段，接下來要把它整合進實際網站並補上後端。

## 現有檔案

- `index.html` — 頁面結構（只留 HTML，不含樣式/邏輯），Fabric.js 用 CDN `<script>` 引入
  （未走 npm 打包），可直接用瀏覽器開啟測試。已推送至 GitHub（`yiyen1030/custom-cloth-print`），
  並串接 Netlify 自動部署（push 到 `main` 即觸發）。
- `style.css` — 所有樣式（CSS 變數、版面、RWD 中斷點）。
- `script.js` — 所有互動邏輯（Fabric.js canvas 初始化、上傳/拖曳/限制範圍、響應式縮放、
  後台可印刷範圍編輯模式）。原本都寫在 `index.html` 的 `<script>` 內，檔案太長不好閱讀才拆出來，
  純搬移沒有改邏輯。
- `assets/polo-shirt-eyes.png` — Polo 衫正面商品圖（800×800，白底線稿），`script.js` 用
  `fabric.Image.fromURL()` 載入並縮放置入 canvas。之後要換其他商品/角度，替換這個檔案即可。

## 技術選型與原因

- **Fabric.js 5.3.0**：用來做「圖片上傳後可拖拉/縮放/旋轉，並限制在指定範圍內」的 canvas 互動。
  選它是因為社群成熟、API 直接、能匯出物件座標，不需要自己刻手勢與變形邏輯。
- 目前用 CDN `<script>` 引入，尚未走 npm/打包流程 — 若專案要整合進正式前端（React/Vue 等），
  建議改用 `npm install fabric` 並依框架封裝成元件。

## 核心邏輯（重要，之後改動要維持這個原則）

1. **前端只用畫布像素（canvas px），完全不做 cm 換算**：`AREA`、上傳圖案的座標/尺寸，全部都是
   `canvas`（480×550px）內的像素座標，程式裡不存在任何「Npx = 1cm」這類寫死的比例尺常數。
   原因：canvas 只是固定尺寸的畫布，跟「這件商品實際多大」沒有內在關係——不同商品、不同拍攝角度，
   同樣的像素範圍對應的實體尺寸都不一樣，在前端假裝算出「公分」只是製造假精度、誤導使用者。
   真正的 px→cm 換算，必須由後端拿到「這個商品這張圖」實際量測過的物理尺寸後才能計算，
   不可以在前端引入任何比例尺常數。
2. **可印刷範圍（AREA）**：一個矩形區域（單位是畫布像素），代表商品上實際能印刷的範圍。
   - 圖案物件的移動/縮放都會被 `constrainToArea()` 限制在這個範圍內，不會超出邊界。
   - 不同商品、不同角度（正面/背面/袖子）都需要各自一組 `AREA` 座標，目前程式只寫了「Polo 衫正面」一組。
3. **輸出資料格式**：按下「產生列印資料」會輸出 JSON，欄位為：
   ```json
   {
     "order_ref": "...",
     "garment": "tshirt",
     "view": "front",
     "canvas_px": { "width": 480, "height": 550 },
     "print_area_px": { "left": 0, "top": 0, "width": 0, "height": 0 },
     "design": {
       "filename": "...",
       "position_px": { "x": 0, "y": 0 },
       "size_px": { "width": 0, "height": 0 },
       "rotation_deg": 0
     }
   }
   ```
   所有座標/尺寸單位都是「畫布像素」，附上 `canvas_px` 當參考框架，方便後端做等比例換算。
   這份 JSON 是前後端的合約，後端要用同一份定義去解析，並自行負責 px→cm 的換算。
4. **響應式縮放不能改動邏輯座標**：`canvas` 的邏輯尺寸固定在 480×550（`AREA`、`position_px`、
   `canvas_px` 都以此為準），畫面上顯示多大只是視覺縮放，由 `fitCanvasToContainer()` 依容器
   可用寬度呼叫 `canvas.setZoom()` + `setDimensions()` 調整。之後如果要改版面、改斷點，
   千萬不要直接改 `<canvas>` 的 `width`/`height` 屬性去做響應式，那會連帶改變所有物件的
   邏輯座標，跟輸出 JSON 的意義對不上；要縮放畫面就只透過 `fitCanvasToContainer()`。
   另外 `.stage`／`.panel` 這類 CSS Grid 欄位都需要 `min-width:0`，否則欄寬會被固定尺寸的
   canvas 撐開，導致手機版整頁橫向溢出（已踩過這個坑）。
5. **後台可印刷範圍編輯模式**：「編輯可印刷範圍」按鈕會讓 `areaRect` 從靜態框變成可選取/
   拖曳/縮放的物件（沿用 Fabric 原生控制點，見 `editAreaBtn`／`saveAreaBtn`／`cancelAreaBtn`
   事件邏輯）。編輯中用 `constrainToCanvas()` 把範圍限制在畫布邊界內（跟限制設計圖案用的
   `constrainToArea()` 是同一套邏輯，只是邊界換成整個 canvas）。儲存時要把 `areaRect` 的
   `scaleX/scaleY` 正規化寫回 `width/height`（見 `saveAreaBtn` handler），否則之後拖曳會疊加
   縮放、算出來的 px 會跟視覺不符。目前只有單一商品、只輸出 JSON 給人工複製，還沒接後端儲存
   （見待辦 #2）。
6. **商品圖必須滿版填滿 canvas，不能留白，而且是自動偵測，不是寫死常數**：`assets/polo-shirt-eyes.png`
   原圖 800x800 四周有留白，`script.js` 的 `detectContentBBox()` 會把圖片畫到暫存 canvas 上、
   用 `getImageData()` 掃描像素，自動找出「非透明、非近白背景」的實際內容邊界（目前這張圖算出
   來是 624x715），再用這個邊界裁切、等比縮放滿版鋪滿整個 canvas（0,0 到畫布邊界完全對齊，
   沒有多餘空白），canvas 高度也是照這個偵測結果的比例自動算出來的（寬度固定 `BASE_CANVAS_WIDTH`
   480，配合版面設計）。
   這樣做的原因：canvas 座標要能當成唯一、可靠的參考基準——如果圖片在畫布裡有留白/置中對齊，
   `position_px`／`AREA` 這些座標的意義就會依附在「圖片在畫布裡的偏移量」這個額外資訊上，之後要
   在別的畫面（例如後台確認畫面）重現同一份定位結果，就必須額外知道這個偏移量才能算對；讓圖片
   滿版對齊 canvas 後，任何消費這份 JSON 的畫面只要「用同一張圖 + 同樣的滿版繪製方式」就能直接
   重現，不用另外換算。
   **之所以做成自動偵測而不是像先前那樣手動跑分析腳本寫死常數**：實際營運換商品圖時不會有人
   （或 AI）重新計算、改程式碼——換掉 `assets` 底下的圖檔、重新整理頁面就會自動抓新圖的邊界、
   自動調整 canvas 比例，不用碰 `script.js`。但 `AREA`（可印刷範圍要放在哪）沒辦法這樣自動判斷
   ——純像素分析看不懂「這是鈕扣」「這是下擺」，換圖後 `setAreaDefaults()` 只會給一個置中的
   通用預設值，實際位置還是要透過「後台編輯可印刷範圍」手動微調一次（見上方第 5 點）。

## 已知限制（暫不處理，但要讓下一個人知道）

- 目前的「合成預覽」只是平面疊圖，沒有模擬衣服皺褶/光影，如果之後要做擬真預覽，
  是另外一條技術路線（例如額外接 AI 圖像生成 API 做視覺效果，跟這裡的精確定位邏輯是分開的兩件事，
  AI 生成圖不能拿來當送印檔，只能當作展示用預覽圖）。
- 沒有多語系、沒有無障礙（a11y）測試。
- 沒有單元測試。
- `detectContentBBox()` 判斷「內容邊界」的邏輯，假設商品圖是「白底/透明底 + 黑色輪廓線」的
  技術平面圖／線稿風格（目前用的 `polo-shirt-eyes.png` 就是這種）。它是靠找出非白、非透明的
  像素（也就是輪廓線本身）來定邊界，不是真的辨識「衣服在哪」。這個假設在線稿風格下永遠成立
  ——不管衣服填色是什麼，一定會有黑色輪廓線把形狀勾出來，人眼才看得懂，程式也才抓得到邊界。
  但如果之後換成**去背實拍照片**（尤其是白色/淺色商品照），輪廓線可能很淡或不存在，這個
  判斷方式就會失效，需要換一套做法（例如改成比對圖片四角顏色當背景參考色，而不是寫死判斷
  「是否接近白色」）。

## 開發慣例

- `index.html`（結構）／`style.css`（樣式）／`script.js`（邏輯）已經分成三個檔案，但都還是
  全域變數、單一 `<script>` 依序執行，尚未模組化。若要擴充功能，建議先決定前端框架（是否導入
  React/Vue）與建置工具（Vite 等），再把 `script.js` 拆成元件，而不是繼續在單檔裡疊加。
- 顏色、字體等視覺樣式目前直接寫在 `style.css` 的 CSS 變數裡（`--paper`、`--ink`、`--thread` 等），
  之後如果有品牌設計系統（design tokens），這裡應該替換成對應的變數來源。
